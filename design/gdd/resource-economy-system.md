# Resource Economy System (资源经济系统)

> **Status**: Draft Complete
> **Author**: user + agents
> **Last Updated**: 2026-05-31
> **Implements Pillar**: Pillar 3 (Build-Up to Payoff — 资源积累支撑后期爆发)

## Overview

资源经济系统管理游戏中唯一的资源——粮草。粮草是所有军事行动的燃料：招募武将、补充兵种、恢复疲劳武将、获取计策卡。系统通过INCOME阶段发放收入（基于领地数量），通过各种消费口制造"我不能同时做所有事"的取舍压力。设计目标是类似TFT金币的简洁：一种资源，多个用途，玩家的策略体现在分配上。

## Player Fantasy

"这回合我收了80粮草，但我既想招一个传奇武将(60)又想补满关羽的骑兵(30)——不够！我必须选择：是先扩军，还是先强化现有阵容？"

单一资源制造的是**分配焦虑**——你总是想做的比能做的多。这就是策略。

## Detailed Design

### Core Rules

#### 1. Resource Definition (资源定义)

| 属性 | 值 |
|------|-----|
| **名称** | 粮草 |
| **类型** | 整数，无上限 |
| **初始值** | 由剧本定义（MVP默认=100） |
| **回合收入** | 基于控制区域数 |
| **消费场景** | 招募武将、补充兵种、恢复武将、获取计策 |

#### 2. Income (收入)

每回合INCOME阶段自动发放：

```
turn_income = base_income + Σ(region[i].income_value)
```

| 收入来源 | 说明 |
|----------|------|
| `base_income` | 固定基础收入（保证最低可操作性） |
| 区域产出 | 每个控制区域有income_value（由大地图系统定义） |

**区域收入参考值**：

| 区域类型 | income_value | 说明 |
|----------|--------------|------|
| 普通区域 | 10-15 | 大多数区域 |
| 富饶区域 | 20-30 | 关键粮仓/商路（争夺焦点） |
| 贫瘠区域 | 5-8 | 战略要地但经济价值低 |

#### 3. Expenses (支出)

| 消费项 | 费用范围 | 说明 |
|--------|----------|------|
| 招募武将(Common) | 20-30 | 从可用将池中招募 |
| 招募武将(Rare) | 40-60 | |
| 招募武将(Legendary) | 80-120 | 重大投资决策 |
| 补充通用兵种 | 5-10/张 | 每回合可补充 |
| 招募精锐兵种 | 20-35/张 | 需要特定条件 |
| 恢复疲劳武将 | 15-30 | 加速恢复（不花则自然恢复需2回合） |
| 获取计策卡 | 15-25 | 从可用计策池中购买 |

#### 4. Economy Pace (经济节奏)

设计目标：每回合收入约覆盖1.5-2个主要动作的费用，确保玩家不能同时做所有事。

```
target: turn_income ≈ 1.5 × average_major_action_cost
```

前期（3-5区域）：收入≈50-70，能做1-2件事
中期（8-12区域）：收入≈100-150，能做2-3件事
后期（15+区域）：收入≈200+，几乎能做所有想做的事（但此时需管理的Stack也更多）

#### 5. No Maintenance Cost (无维护费)

**设计决策**：已有的兵种和武将不消耗每回合粮草。粮草只在"做新事情"时消耗。

理由：
- 维护费会惩罚扩张（与征服玩法矛盾）
- TFT模式：棋子上场后不花钱，钱只用来买新棋子
- 简化认知负担——玩家只需思考"我要买什么"而非"我养不养得起"

---

### States and Transitions

资源系统状态简单：

```
grain: int (当前粮草数)

状态变化仅两种：
  +income (INCOME阶段)
  -cost   (玩家执行消费操作时)
```

无复杂状态机。粮草不会过期、不会腐烂、不会被掠夺（MVP）。

### Interactions with Other Systems

| 系统 | 交互方式 |
|------|----------|
| **回合流程系统** | INCOME阶段信号 → 触发收入结算 |
| **大地图系统** | 读取各区域的income_value |
| **武将管理系统** | 招募武将时扣除粮草 |
| **兵种管理系统** | 补充/招募兵种时扣除粮草 |
| **计策系统** | 获取计策卡时扣除粮草（Future: Vertical Slice） |

## Formulas

### 回合收入

```
turn_income = BASE_INCOME + Σ(controlled_regions[i].income_value)

示例（中期，控8区域）：
turn_income = 20 + (15+15+10+20+10+12+15+25) = 20 + 122 = 142
```

### 消费校验

```
can_afford(cost) → bool:
  return current_grain >= cost

execute_purchase(cost):
  assert can_afford(cost)
  current_grain -= cost
```

### 经济平衡锚点

```
legendary_general_cost = ~2回合积蓄（前期）/ ~1回合积蓄（后期）
full_stack_from_scratch = ~3-4回合积蓄（前期）

这确保"存钱买传奇"是一个有意义的战略选择（忍耐2回合不花钱）
```

## Edge Cases

| # | 边界情况 | 处理规则 |
|---|----------|----------|
| 1 | 粮草为0且无法招兵 | 允许。玩家仍可用现有Stack战斗。下回合INCOME会恢复收入（只要还有区域） |
| 2 | 失去所有产粮区域 | base_income保证最低收入（20/回合），但极度困难——触发"困境"体验 |
| 3 | 粮草溢出（收入远大于支出） | 无上限，允许积累。后期玩家可能有大量粮草——这是"称霸"的正确体感 |
| 4 | 玩家在DEPLOY阶段花粮草（招兵在什么时机？） | 招募/补充操作在DEPLOY阶段执行（和组Stack同一时间）。花粮草是DEPLOY阶段的一部分 |
| 5 | 中途丢失区域导致收入锐减 | 即时生效：下回合INCOME按当前控制区域计算。失去富饶区域是真实的经济打击 |
| 6 | 费用不足时的UI处理 | 不可点击/灰色按钮 + 提示"还需X粮草"。不允许负数。 |

## Tuning Knobs

| 参数名 | 当前值 | 安全范围 | 影响 |
|--------|--------|----------|------|
| `BASE_INCOME` | 20 | 10-40 | 保底收入。太高→失去区域无痛感；太低→逆境太绝望 |
| `STARTING_GRAIN` | 100 | 50-200 | 初始粮草。决定第1回合的自由度 |
| `GENERAL_COST_LEGENDARY` | 100 | 60-150 | 传奇武将价格。决定"存钱vs花钱"的张力 |
| `TROOP_COMMON_COST` | 8 | 5-15 | 通用兵种价格。太低→兵力过于廉价 |
| `TROOP_ELITE_COST` | 25 | 15-40 | 精锐兵种价格 |
| `RECOVERY_COST` | 20 | 10-40 | 武将恢复加速费用 |
| `INCOME_PER_REGION_AVG` | 15 | 8-25 | 平均每区域产出。决定扩张的经济激励强度 |

## Visual/Audio Requirements

- INCOME阶段：粮草数字跳动增长动画 + 硬币音效
- 消费时：粮草数字减少 + 轻微"支出"音效
- 余额不足时：数字闪红

## UI Requirements

- 屏幕上方常驻显示当前粮草数
- INCOME阶段：显示收入明细（"基础+20, 荆州+15, 长安+25 = +60"）
- DEPLOY阶段：所有需花粮草的操作旁显示费用；余额不足的选项灰色
- 鼠标悬停粮草数字：显示下回合预期收入
