# Battle Report System (战报系统)

> **Status**: Draft Complete
> **Author**: user + agents
> **Last Updated**: 2026-06-01
> **Implements Pillar**: Pillar 3 (Build-Up to Payoff — 战报是"构筑回报"的兑现时刻)

## Overview

战报系统将自动战斗系统输出的BattleEvent数组渲染为可读的文字战报。战报是玩家构筑策略的"回报兑现"——看到"关羽·突破触发→骑兵追击→决定性胜利"就是整个游戏循环的高潮。战报是**可选深度**的：默认显示胜负结果标签，玩家想看详情时展开逐轮文字回放。不强制弹窗，不打断节奏。

## Player Fantasy

"打完了——胜利！我点开战报标签，看到第1轮周瑜火计烧了40HP、第3轮敌方溃败的全过程。这比我预想的还完美。截图发群里。"

战报是可选的快感，不是强制的信息噪音。

## Detailed Design

### Core Rules

#### 1. Report Display Mode (展示模式 — 关键设计决策)

**战报不是弹窗，是可收起的标签。**

| 展示层次 | 内容 | 交互 |
|----------|------|------|
| **结果标签** | "胜利/失败/平局" + 区域名 | 始终显示在REPORT阶段 |
| **摘要** | 轮数 + 关键技能触发 + 伤亡概览 | 点击标签展开 |
| **详细回放** | 逐轮文字战报，每轮的伤害/技能/状态变化 | 在摘要内点击"查看详情" |

**可选深度原则**（来自原型测试结论）：
- 胜利战报：玩家更想看（"看我组合多强"）
- 失败战报：玩家想知道为什么输（"哪里出了问题"）
- 玩家永远不会被强制阅读战报

#### 2. Report Data Source (数据来源)

战报系统纯渲染——不做任何计算：

```
输入: BattleResult (from 自动战斗系统)
  - winner
  - rounds_fought
  - events: array[BattleEvent]
  - casualties

输出: 格式化文字战报 (分层渲染)
```

#### 3. Report Text Template (文字模板)

**结果标签**：
```
"[胜利] 襄阳之战 — 关羽军 击败 张角军 (2轮)"
```

**摘要**：
```
【襄阳之战】
攻方: 关羽军 (战力135) vs 守方: 张角军 (战力80)
地形: 城池
战斗轮数: 2轮

关键事件:
  ✦ 关羽·突破 触发 — 造成40额外伤害
  ✦ 骑兵获得平原地利 +10%

结果: 攻方胜利
伤亡: 攻方损失HP 80 | 守方全灭（张角疲劳，步兵×3覆灭）
```

**详细回放**：
```
— 第0轮 (战前) —
  地形标签注入: 攻方获得「平原地利」
  
— 第1轮 —
  ✦ [技能] 关羽·突破 触发! (条件: 骑兵≥2, 满足) → 守方 -40HP
  攻方攻击: 140 × 1.03 = 144 → 守方HP: 300→116
  守方攻击: 80 × 0.97 = 78 → 攻方HP: 450→372

— 第2轮 —
  攻方攻击: 140 × 1.08 = 151 → 守方HP: 116→0 ← 溃败!
  守方攻击: 80 × 0.92 = 74 → 攻方HP: 372→298

— 结果 —
  ★ 攻方胜利 (2轮)
  攻方剩余HP: 298/450
  守方伤亡: 张角(疲劳), 步兵×3(覆灭)
```

#### 4. Report Collection (战报收藏)

- 所有战报保存在当局游戏中，可随时回看
- REPORT阶段的列表按回合分组展示
- 未来可增加"收藏"功能（截图/分享）

---

### States and Transitions

无状态系统——纯渲染函数：

```
render_report(battle_result) → ReportView {
  tag: string       # 结果标签
  summary: string   # 摘要文本
  detail: string    # 详细回放文本
}
```

### Interactions with Other Systems

| 系统 | 交互方式 |
|------|----------|
| **自动战斗系统** | 接收BattleResult数据 |
| **回合流程系统** | REPORT阶段展示战报列表 |

## Formulas

无计算公式。战报系统是纯文本渲染。

唯一的"公式"是文本格式化规则：
```
技能触发行 = "✦ [技能] {general_name}·{skill_name} 触发! → {effect_description}"
伤害行 = "{side}攻击: {power} × {variance} = {actual} → {target}HP: {old}→{new}"
```

## Edge Cases

| # | 边界情况 | 处理规则 |
|---|----------|----------|
| 1 | 战斗在pre_battle阶段就结束（一击必杀） | 显示简短战报："[技能]一击致命"，无combat rounds记录 |
| 2 | 平局（8轮后双方都存活） | 战报显示"双方僵持不下，攻方撤退" |
| 3 | 本回合无战斗 | REPORT阶段显示"本回合无战事"，玩家直接点"下一回合" |
| 4 | 大量战斗同时发生（5+区域） | 按区域列表展示标签，玩家选择性展开。不自动展开所有 |
| 5 | 战报文本过长（8轮战斗） | 详细回放可滚动，摘要始终保持在一屏内 |

## Dependencies

**上游依赖**：

| 系统 | 依赖内容 |
|------|----------|
| 自动战斗系统 | BattleResult + BattleEvent数组 |

**下游依赖方**：无（终端展示系统）

## Tuning Knobs

| 参数名 | 当前值 | 安全范围 | 影响 |
|--------|--------|----------|------|
| `DEFAULT_REPORT_EXPANDED` | false | bool | 默认是否展开战报。false=只显示标签 |
| `HIGHLIGHT_SKILL_TRIGGERS` | true | bool | 技能触发行是否高亮显示 |
| `SHOW_EXACT_NUMBERS` | true | bool | 是否显示精确数字（false=只显示"重创/轻伤"等描述词） |
| `MAX_REPORTS_STORED` | 当局全部 | 20-无限 | 保留多少回合的历史战报 |

## Visual/Audio Requirements

- 结果标签：胜利=金色，失败=暗红，平局=灰色
- 技能触发行用特殊颜色高亮（金色文字+星标图标）
- 展开/收起战报有平滑动画
- 关键伤害数字用加大字号显示

## UI Requirements

- REPORT阶段左侧：战报标签列表（按区域排列）
- 点击标签→右侧展开摘要
- 摘要底部"查看详情"→展开逐轮回放
- 底部常驻"下一回合"按钮
- 历史战报可在非战斗阶段通过菜单查看

## Acceptance Criteria

| # | 验收条件 | 验证方式 |
|---|----------|----------|
| 1 | 战报正确渲染所有BattleEvent | 集成测试：预设战斗→验证战报文本 |
| 2 | 三层展示（标签→摘要→详情）正常工作 | 手动测试 |
| 3 | 无战斗回合显示"无战事" | 单元测试 |
| 4 | 技能触发正确高亮 | 手动测试 |
| 5 | 多战斗按列表展示，不自动全部展开 | 手动测试 |
| 6 | 战报文本中数字与实际战斗计算一致 | 集成测试 |
